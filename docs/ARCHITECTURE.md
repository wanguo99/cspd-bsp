# 系统架构设计

## 系统概述

本框架专为**卫星算存载荷管理板**设计，作为卫星平台与算存载荷之间的桥接设备。

```
┌─────────────┐                    ┌──────────────┐                    ┌─────────────┐
│  卫星平台    │ <--- CAN 2.0 ---> │   管理板      │ <--- Ethernet --> │  算存载荷    │
│ (自定义协议) │                    │  (本框架)     │ <--- UART(备份)-> │ (IPMI/Redfish)│
└─────────────┘                    └──────────────┘                    └─────────────┘
```

## 核心设计原则

1. **稳定性第一**: 看门狗、故障恢复、通信冗余
2. **低耦合**: 模块间通过消息队列解耦
3. **可维护**: 清晰的模块划分和日志系统
4. **可扩展**: 预留MCU移植接口

## 分层架构

```
┌─────────────────────────────────────────────────────────┐
│                    应用层 (Apps)                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │CAN网关   │  │协议转换  │  │状态监控  │  │故障处理 │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────┘ │
├─────────────────────────────────────────────────────────┤
│              硬件抽象层 (HAL)                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │CAN驱动   │  │网络驱动  │  │串口驱动  │              │
│  └──────────┘  └──────────┘  └──────────┘              │
├─────────────────────────────────────────────────────────┤

└─────────────────────────────────────────────────────────┘
```

## 核心模块说明

### 1. OSAL (操作系统抽象层)

提供统一的操作系统接口，支持跨平台移植。

**核心模块**:
- **任务管理** (os_task.c): 任务创建、删除、延时、优先级管理
- **消息队列** (os_queue.c): 模块间异步通信
- **互斥锁** (os_mutex.c): 共享资源保护
- **时间服务** (os_clock.c): 时间获取、滴答计数
- **内存管理** (os_heap.c): 堆内存信息查询
- **错误处理** (os_error.c): 错误码转换
- **日志系统** (os_log.c): 分级日志输出

**设计特点**:
- 接口统一，易于移植到FreeRTOS等RTOS
- 基于pthread实现，性能优异
- 环形缓冲区消息队列，无动态内存分配

### 2. HAL (硬件抽象层)

封装硬件相关操作，隔离平台差异。

**核心驱动**:
- **CAN驱动** (hal_can_linux.c): 基于SocketCAN实现
- **载荷通信** (hal_server_linux.c): 以太网+UART双通道

**设计特点**:
- 统一的驱动接口
- 支持通信通道自动切换
- 故障检测和恢复机制

### 3. Apps (应用层)

实现具体业务逻辑。

**核心应用**:
- **CAN网关** (can_gateway): 
  - 接收卫星平台CAN消息
  - 解析自定义CAN协议
  - 转发到协议转换模块
  - 发送响应和状态上报

- **协议转换** (protocol_converter):
  - 卫星命令 → IPMI/Redfish命令
  - 载荷响应 → 卫星协议格式
  - 支持命令队列和超时处理

**设计特点**:
- 模块间通过消息队列通信
- 独立的任务，互不阻塞
- 易于添加新应用模块

## 数据流示例

### 命令转发流程

```
卫星平台 --[CAN]-->
  CAN网关 --[MSG_QUEUE]-->
    协议转换 --[Ethernet/UART]-->
      载荷 --[响应]-->
        协议转换 --[MSG_QUEUE]-->
          CAN网关 --[CAN]-->
            卫星平台
```

### 状态监控流程

```
定时器触发 -->
  状态监控 --[IPMI/Redfish]-->
    载荷 --[状态数据]-->
      状态监控 --[MSG_QUEUE]-->
        CAN网关 --[CAN]-->
          卫星平台
```

## 可扩展性

### 预留接口

1. **MCU移植**: OSAL接口统一，只需实现FreeRTOS版本
2. **新协议**: 协议转换模块易于扩展
3. **新应用**: 应用模块独立，易于添加
4. **新命令**: CAN协议定义清晰，易于扩展

## 故障处理机制

### 以太网故障

1. 检测连续5次通信失败
2. 自动切换到UART通道
3. 定期尝试恢复以太网
4. 恢复后自动切回以太网

### 载荷离线

1. 返回离线状态码给卫星平台
2. 缓存命令等待载荷恢复
3. 定期重试连接
4. 恢复后执行缓存命令

### 任务异常

1. 软件看门狗监控任务心跳
2. 检测到异常后尝试重启任务
3. 多次失败后触发系统重启
4. 记录异常日志

## 开发建议

### 代码规范

- 使用C99标准
- 函数名使用大驼峰: `OS_TaskCreate`
- 变量名使用小写+下划线: `task_id`
- 宏定义使用全大写: `MAX_TASKS`

### 错误处理

```c
int32 ret = OS_TaskCreate(...);
if (ret != OS_SUCCESS) {
    OS_printf("错误: %s\n", OS_GetErrorName(ret));
    // 处理错误
}
```

### 日志使用

```c
OS_printf("[模块名] 信息\n");
// 日志会自动添加时间戳
```
