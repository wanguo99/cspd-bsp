# PMC-BSP 软件架构优化方案

## 1. 项目概述

### 1.1 项目背景
PMC-BSP（卫星算存载荷管理控制器板级支持包）是一个面向航空航天领域的嵌入式软件平台，为卫星载荷管理控制器提供底层硬件抽象和外设驱动支持。

### 1.2 当前架构概览
项目采用5层软件架构设计：
- **OSAL层**：操作系统抽象层，提供跨平台的OS服务
- **HAL层**：硬件抽象层，封装底层驱动
- **XConfig层**：外设配置管理层，仅用于PDL层使用，支持多平台配置
- **PDL层**：外设驱动库，提供MCU/卫星/BMC外设接口
- **Apps层**：应用层，实现CAN网关和协议转换等功能

### 1.3 代码规模统计
- 总代码量：约18,000行
- 生产代码：约14,000行
- 测试代码：约4,000行
- 代码文件：97个（.c/.h）
- 模块数量：5层架构（OSAL/HAL/XConfig/PDL/Apps）

## 2. 架构现状分析

### 2.1 优势分析

#### 2.1.1 清晰的分层架构
- 层次职责明确，依赖关系单向（OSAL→HAL→PDL→Apps）
- 模块化设计良好，便于维护和扩展
- 配置下沉到各模块，降低耦合度

#### 2.1.2 良好的可移植性
- OSAL层提供统一的OS接口抽象
- XConfig支持多平台/多产品/多版本配置
- 构建系统支持交叉编译和本地编译

#### 2.1.3 健壮的设计模式
- 任务优雅关闭机制（OS_TaskShouldShutdown）
- 通信冗余设计（以太网主通道+UART备份）
- 统计信息使用原子操作保证线程安全
- 错误处理机制完善（OS_Error模块）

#### 2.1.4 完善的测试覆盖
- 测试代码占比约28%
- 单元测试覆盖主要模块
- 使用CMocka测试框架

### 2.2 问题识别

#### 2.2.1 架构层面问题

**问题1：XConfig层定位已明确**
- 现状：XConfig层已明确定位为硬件配置库，仅供PDL层使用
- 职责：以外设为单位管理硬件配置，类似Linux设备树
- 架构：两层配置架构（硬件配置层 + APP配置层）
- 状态：✓ 已解决（v1.0.0）

**问题2：HAL层过于薄弱**
- 现状：HAL层仅包含CAN和串口驱动，其他外设直接在PDL层实现
- 影响：硬件抽象不完整，移植到新硬件平台时需要修改PDL层代码
- 建议：扩展HAL层，将所有硬件相关代码下沉到HAL层，包括但不限于UART、CAN、SPI、I2C、GPIO等硬件读写操作

**问题3：PDL层职责已优化**
- 现状：PDL层（Peripheral Driver Layer）定位为外设驱动层
- 设计理念：管理板为核心，卫星/载荷/BMC/MCU统一抽象为外设
- 职责：提供外设操作接口，可包含简单的协议封装
- 状态：✓ 已优化（v1.0.0，Service层重命名为PDL层）

#### 2.2.2 代码质量问题

**问题4：头文件组织不统一**
- 现状：部分模块使用三层头文件（api/internal/peripheral），部分模块仅有单层头文件
- 影响：接口暴露不一致，增加使用难度
- 建议：统一头文件组织规范，明确公开接口和内部接口

**问题5：错误处理不一致**
- 现状：部分函数返回错误码，部分函数使用OS_Error记录错误
- 影响：错误处理方式不统一，增加调试难度
- 建议：制定统一的错误处理规范

**问题6：日志系统功能有限**
- 现状：OS_Log模块功能简单，缺少日志级别过滤、日志轮转等功能
- 影响：调试和故障排查效率低
- 建议：增强日志系统功能，支持动态日志级别、日志文件轮转、远程日志等

#### 2.2.3 性能和资源问题

**问题7：内存管理缺少监控**
- 现状：OS_Heap模块提供内存分配，但缺少内存使用统计和泄漏检测
- 影响：难以发现内存泄漏和内存碎片问题
- 建议：增加内存使用统计、内存池机制、内存泄漏检测

**问题8：缺少性能监控机制**
- 现状：无系统级性能监控，难以发现性能瓶颈
- 影响：优化工作缺少数据支撑
- 建议：增加性能监控模块，统计任务CPU占用、消息队列深度、中断延迟等指标

#### 2.2.4 可维护性问题

**问题9：文档不完善**
- 现状：缺少架构设计文档、接口文档、移植指南等
- 影响：新成员上手困难，维护成本高
- 建议：补充完善文档体系

**问题10：缺少持续集成**
- 现状：无自动化构建和测试流程
- 影响：代码质量难以保证，回归测试成本高
- 建议：建立CI/CD流程，自动化构建、测试、代码检查

## 3. 优化方案

### 3.1 架构优化

#### 3.1.1 重新定义层次职责

**OSAL层（操作系统抽象层）**
- 职责：提供跨平台的OS服务抽象
- 包含：任务、队列、互斥锁、信号量、时钟、堆、文件、网络、日志、错误处理
- 原则：不包含任何硬件相关代码，不包含业务逻辑

**HAL层（硬件抽象层）**
- 职责：封装所有硬件相关代码，提供统一的硬件接口
- 包含：GPIO、UART、CAN、SPI、I2C、ADC、PWM、定时器等驱动
- 原则：仅提供硬件操作接口，不包含业务逻辑

**XConfig层（配置管理层）**
- 职责：管理硬件配置信息，支持多平台/多产品/多版本配置
- 包含：配置注册、配置查询、配置选择逻辑
- 原则：仅负责配置管理，不包含业务逻辑

**PDL层（外设驱动库）**
- 职责：基于HAL层提供高级外设操作接口
- 包含：MCU外设、卫星接口、BMC接口等
- 原则：提供外设操作接口，可包含简单的协议封装，但不包含复杂业务逻辑

**Apps层（应用层）**
- 职责：实现具体的业务功能
- 包含：CAN网关、协议转换器、数据采集、遥测遥控等应用
- 原则：基于PDL层实现业务逻辑，不直接访问HAL层

#### 3.1.2 HAL层扩展计划

**阶段1：补充基础驱动**
- 增加GPIO驱动（hal_gpio.c/h）
- 增加SPI驱动（hal_spi.c/h）
- 增加I2C驱动（hal_i2c.c/h）
- 增加定时器驱动（hal_timer.c/h）

**阶段2：统一驱动接口**
- 定义统一的驱动初始化接口（HAL_XXX_Init）
- 定义统一的驱动配置接口（HAL_XXX_Config）
- 定义统一的驱动操作接口（HAL_XXX_Read/Write/Control）
- 定义统一的驱动错误处理机制

**阶段3：支持多平台**
- 为每个驱动提供多平台实现（TI AM62/STM32/Linux等）
- 通过编译选项选择具体平台实现
- 确保接口在所有平台上保持一致

#### 3.1.3 PDL层重构计划

**重构目标：**
- 将业务逻辑上移到Apps层
- PDL层仅提供外设操作接口
- 简化PDL层代码，提高可复用性

**重构步骤：**
1. 分析当前PDL层代码，识别业务逻辑部分
2. 将业务逻辑提取到Apps层
3. 重构PDL层接口，使其更加通用
4. 更新测试用例

**示例：卫星通信模块重构**
- 当前：pdl_satellite模块包含通信协议解析和业务处理
- 重构后：
  - PDL层：提供卫星接口的读写操作（PDL_Satellite_Send/Receive）
  - Apps层：实现协议解析和业务处理（SatelliteProtocolHandler）

### 3.2 代码质量优化

#### 3.2.1 统一头文件组织规范

**规范定义：**
- 每个模块提供三层头文件：
  - `xxx.h`：公开API接口，供外部模块使用
  - `xxx_internal.h`：内部接口，供模块内部使用
  - `xxx_types.h`：类型定义，供公开接口和内部接口共享

**实施步骤：**
1. 为所有模块创建三层头文件
2. 将公开接口移到xxx.h
3. 将内部接口移到xxx_internal.h
4. 将类型定义移到xxx_types.h
5. 更新所有#include语句

#### 3.2.2 统一错误处理规范

**规范定义：**
- 所有函数返回错误码（OS_Status_t）
- 使用OS_Error模块记录详细错误信息
- 定义统一的错误码体系

**错误码体系：**
```c
typedef enum {
    OS_SUCCESS = 0,
    OS_ERROR_INVALID_PARAM = -1,
    OS_ERROR_NO_MEMORY = -2,
    OS_ERROR_TIMEOUT = -3,
    OS_ERROR_NOT_FOUND = -4,
    OS_ERROR_ALREADY_EXISTS = -5,
    OS_ERROR_NOT_INITIALIZED = -6,
    OS_ERROR_HARDWARE_FAILURE = -7,
    // ... 更多错误码
} OS_Status_t;
```

**使用示例：**
```c
OS_Status_t PDL_Satellite_Send(const uint8_t *data, size_t len) {
    if (data == NULL || len == 0) {
        OS_Error_Set(OS_ERROR_INVALID_PARAM, "Invalid parameters");
        return OS_ERROR_INVALID_PARAM;
    }
    
    // ... 发送数据
    
    if (send_failed) {
        OS_Error_Set(OS_ERROR_HARDWARE_FAILURE, "Satellite send failed");
        return OS_ERROR_HARDWARE_FAILURE;
    }
    
    return OS_SUCCESS;
}
```
